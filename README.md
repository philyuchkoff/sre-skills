## sre-guru — это **консультант**

```
Пользователь: "Как оптимизировать Kubernetes?"
Скилл: задаёт вопросы → даёт рекомендации → ответ в чате
```
SRE, которого можно позвать на созвон.

## sre-auditor — **автоматизированный аудитор с отчётностью**

```
Пользователь: "SRE audit вот этот k8s манифест"
Скилл: 
1. Задаёт 3 вопроса про RPS, SLO, стек
2. Проверяет по чек-листам (их можно поправить под себя)
3. Находит: нет Pod Disruption Budget, нет topology spread, нет readinessProbe
4. Классифицирует: 🔴 Critical (найти за 24 часа)
5. Пишет `SRE.md` в корень проекта
6. Пишет в чате: "Файл создан, начинайте с пунктов Critical"
```

SRE, который оставляет **письменный аудит** и требует исправлений.

1. Генерирует файл `SRE.md`, чтобы следы аудита остались в директории
2. Приоритеты с дедлайнами: бизнес понимает, что горит (Шаг 3: Классификация рисков по приоритету)
3. Не использует в рекомендациях платные сервисы, юридически безопасно для РФ, только OpenSource, то есть DataDog предлагать не будет.
4. Рекомендует команды для проверки, которые можно выполнить немедленно
5. Указывает "слабое место аудита", честный перед пользователем ("Без реального прода и нагрузки я не могу проверить..."
   
## Сравнительная таблица

| Характеристика | 'sre-guru' | 'sre-auditor' |
|----------------|----------------|------------------|
| **Основная роль** | Эксперт, отвечающий на вопросы | Инструмент аудита с генерацией артефактов |
| **Режим работы** | Только консультативный | Консультативный + **аудиторский** |
| **Обязательный вывод** | Текстовый ответ в чате | **SRE.md файл** в репозитории |
| **Работа с кодом** | Может обсудить абстрактно | **Формальный анализ** по чек-листам |
| **Приоритеты** | Неявные | 🔴 Critical / 🟡 High / 🔵 Medium / ⚪ Low |
| **OpenSource** | Упомянуто | **Жёсткий запрет** платных сервисов + примеры конфигов |
| **Команды для продуктива** | Нет | `kubectl`, `docker`, `journalctl` — готовые строки |
| **Checklist'ы** | Нет | Для кода, observability, CI/CD, DR |
| **Моделирование отказов** | Нет | Chaos experiment в будущем :) |
| **Trade-offs анализ** | Есть | Есть, усиленный |
| **Три взгляда** | Сторонник / Критик / Прагматик | Добавлен контекст аудита |
| **Слабое место аудита** | Нет | **Обязательный** раздел с честностью |

Разница между ними, как между **"давай я расскажу"** и **"вот отчёт, исправляй Critical за 24 часа"**.

Оба общаются формально, строго, задают несколько уточняющих вопросов перед ответом, проводят trade-offs 
анализ (объясняют **компромиссы** между разными решениями, потому что идеальных решений не существует, например: "Хочешь низкую задержку (latency) - тогда
жертвуешь согласованностью данных (consistency). Хочешь высокую надёжность - тогда жертвуешь скоростью изменений (velocity)"), 
предоставляют три "точки зрения" (сторонник/критик/прагматик), ответы и общение подразумевают, что пользователь может и не быть монстром в SRE (объясняет термины и т.п.)
