## Эксперт по SLO/SLI, по выбору SLI, управлению error budget. Только OpenSource.

Скилл для SRE, который помогает создавать, анализировать и автоматизировать SLO/SLI и управлять error budget.

## 🎯 Что делает скилл

| Режим | Что умеет |
|-------|-----------|
| **Консультация** | Объясняет SLO/SLI, помогает выбрать метрики, рассказывает про burn rate |
| **Генерация SLO** | Создаёт готовый SLO документ в 3 форматах (Markdown/YAML/JSON) |
| **Анализ SLO** | Проверяет существующие SLO на ошибки и даёт рекомендации |

## 📋 Форматы вывода

| Формат | Команда | Когда использовать |
|--------|---------|---------------------|
| **Markdown** | `output_format: markdown` (по умолчанию) | Для согласования с бизнесом |
| **YAML** | `output_format: yaml` | Для GitOps, хранения в репозитории |
| **JSON** | `output_format: json` | Для Terraform, API, CI/CD интеграции |

```yaml

```

## 🔧 Интеграция с мониторингом (JSON формат)

JSON формат совместим с:

- **Terraform** — создание SLO как ресурса
- **Prometheus Operator** — SLO CRD через API
- **VictoriaMetrics / vmalert** — автоматические алерты
- **CI/CD pipelines** — проверка error budget перед деплоем
- **ChatOps (Telegram)** — статус SLO в реальном времени

### Пример: CI/CD проверка перед деплоем

```bash
# Получить статус SLO из API
curl -s https://slo-api.company.com/v1/slos/payment-api | jq . > slo.json

# Проверить остаток бюджета
BUDGET=$(jq '.status.current_error_budget_remaining' slo.json)

if (( $(echo "$BUDGET < 10" | bc -l) )); then
    echo "❌ Error budget < 10% — деплой заблокирован"
    exit 1
fi
```

## 📚 SLI Selection Guide — что измерять

### Request-driven (CRUD API)
| SLI | Формула | Когда нужно |
|-----|---------|-------------|
| Availability | успешные запросы / все запросы | всегда |
| Latency | запросы быстрее порога / все запросы | для синхронных вызовов |
| Quality | запросы без деградации / все запросы | если есть fallback |

### Pipeline-driven (ETL/streaming)
| SLI | Когда нужно |
|-----|-------------|
| Freshness | данные обновлены недавно |
| Correctness | данные прошли валидацию |
| Coverage | обработано X% от всех данных |

### Infrastructure
| SLI | Когда нужно |
|-----|-------------|
| Compute availability | CPU/память не насыщены |
| Dependency availability | внешние API доступны |
| Queue processing | сообщения обработаны до дедлайна |

## 📊 Error Budget Best Practices

### 1. Rolling Window (не календарный месяц)
```
❌ 30 calendar days — сброс 1-го числа, паника в конце месяца
✅ 28 days rolling — плавно, без шоков
```

### 2. Burn Rate алерты (лучше чем пороговые)

| Burn Rate | Время до исчерпания | Действие |
|-----------|---------------------|----------|
| >14 | < 1 час | Page немедленно |
| >6 | ~5 часов | Page через 15 минут |
| >2 | ~17 часов | Ticket |
| >1 | >28 дней | Только логи |

Формула: `burn_rate = error_rate / (1 - slo_target)`

### 3. Action Policy

| Остаток бюджета | Действие |
|-----------------|----------|
| >75% | Обычная работа |
| 25-75% | Усиленный code review |
| <25% | Только фиксы надёжности |
| 0% | Feature freeze |

## 🚫 Частые ошибки

| Ошибка | Почему плохо | Как исправить |
|--------|--------------|---------------|
| SLO = 100% | Нет места для деплоев/инцидентов | 99.9% или 99.99% |
| Нет SLI | Нельзя измерить → бесполезно | Начни с HTTP статусов |
| Календарный месяц | Сброс бюджета → паника | Rolling 28 дней |
| Алерт при нарушении SLO | Уже поздно, всё сломалось | Алерт на burn rate >1 |
| Нет action policy | Никто не реагирует | Опиши чёткие шаги |
